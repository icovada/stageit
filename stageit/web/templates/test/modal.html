Forms

Forms let users interact with the interface and enter any required information. The examples below demonstrate any applicable display options and how to use each type of form control.

Do's

  • Use only one form per page.

Text inputs

Anatomy

Labels

A label helps users understand what type of information they should enter. Use a secondary label style to hint an alternative entry.

Placeholder text

Placeholder text appears within an input field. It should give a hint to users what to enter.

Help text (optional)

Help text should appear below the text field and be persistently visible. They can be used in addition to the placeholder text. Use help text if the hint requires to be visible to users while entering (e.g. a character counter with remaining characters if the input fields has a limited character entry).

Tooltip (optional)

Use tooltips to provide context for complex questions that cannot be covered in a help text, e.g. they could contain definitions for abbreviations used within the label – if the label is GST Number the tooltip could be GST stands for Goods and Services Tax Number.

Inputs

The type of text input determines what kind of characters are allowed inside the field. For mobiles/tablets keyboards may be optimized for more refined inputs (e.g. if user has to input a phone number – number keypad should appear when focused).

Required fields

All fields should be considered mandatory or required. Therefore form labels don’t need to be marked a required fields.

Input restrictions

Restrictions can be applied to text inputs to guide users towards a better interaction. Those restrictions have to be made clear to the user (e.g. ‘Max 300 characters’ for a text input). These restrictions include:

  • disabled – disable a text input to signify something else needs to happen first.
  • max – specify the maximum value for an text input.
  • maxlength – maximum amount of characters for a text input.
  • min – minimum value for a text input.
  • pattern – specifies a certain expression to check against (e.g. phone numbers).
  • readonly – text input cannot be changed.
  • required – mandatory text input.
  • value – default value for an text input.

Examples

Maximum of 60 characters

Do's

  • Always add labels. Screen readers will have trouble with your forms if you don't include a label for every input.
  • Show the appropriate keyboard type.
  • Keep text legible.

Textareas

Textareas allow users to input text, select text, and lookup data via auto-completion.

Single-line text inputs automatically scroll their content to the left as the cursor reaches the right edge of the text input.

Multi-line textareas automatically span multiple lines for overflow text and scroll vertically when the cursor reaches the lower edge.

Auto-Complete

Use auto-complete text inputs to present real-time suggestions or completions in dropdowns, so users can enter information more accurately and efficiently. Autocomplete is especially valuable for users with limited text literacy or who have difficulty with spelling, especially if they are using a non-native language.

Examples

<div class="form-group">
          <label class="control-label" for="exampleTextarea1">Message</label>
          <textarea rows="5" id="exampleTextarea1" class="form-control" placeholder="This is a text area"></textarea>
        </div>

Do's

  • Use a text input to request a small amount of information, such as an email address.
  • Use a text area to request long or multi-line information, such as a postal address.
  • Show a hint in text fields to help communicate purpose.
  • Use secure form fields when appropriate. Always use a secure fields when an app asks for sensitive data, such as a password.
  • Use email fields for email addresses.
  • Space multiple fields evenly.

Dont's

  • Don’t use a separate label to describe a text input on mobiles when placeholder text is sufficient.

Password inputs

The text in a password input is disguised by default. Midline ellipses are displayed to represent each character of a password entered (such as •••••••).

Enabling & disabling visibility of passwords

When the visibility icon is displayed with a text input, it indicates whether or not that field’s input is visible. The text input's visibility may be toggled on or off using the icon.

Examples

Minimum of 6 characters including numerics

        </div>

Checkboxes

Checkboxes are used when you want to let the visitor select one or more options from a set of alternatives. If only one option is to be selected at a time you should use radio buttons instead.

Do's

  • Use checkboxes if users have multiple options to choose.

Dont's

  • If users have to select only one option, radio buttons have to be used.

Radio buttons are shown in a group of two or more. The user can only select one radio from a radio group at a time.

Do's

  • Use Radio buttons if users have to select one option only.
  • Use the same name attribute on all radio buttons in the group.

Dont's

  • If users have to or are allowed to use multiple options then checkboxes must be used.

File Picker

Attaching files to a form are performed by providing a drop area and alternative link for users to choose a source. While a file is being uploaded, the dropzone is disabled to prevent multiple uploads and possible errors. A short animation of success (a green tick icon where the filesize appears) can be signified to users as feedback for a 100% upload. A failed attempt will trigger an error message.

Multiple uploaded attachments are wrapped into an accordion. Revealing those attachments allows users to delete individual files.

Validation

Form validation is the process of making sure that data supplied by the user using a form, meets the criteria set for collecting data from the user.

Bootstrap includes validation styles for error, warning, and success states on form controls. To use, add .has-warning, .has-error, or .has-success to the parent element. Any .form-control, .help-block, and .form-control-feedback within that element will receive the validation styles.

Using these validation styles to denote the state of a form control only provides a visual, colour-based indication, which will not be conveyed to users of assistive technologies - such as screen readers - or to colour-blind users. Ensure that an alternative indication of state is also provided. For instance, you can include a hint about state in the form control's <label> text itself (as is the case in the following code example), include a glyph (with appropriate alternative text using the .sr-only class), or by providing an additional help text block. Specifically for assistive technologies, invalid form controls can also be assigned an aria-invalid="true" attribute.

text inputs can be checked for:

  • Required fields – check if users has entered something.
  • All required letters – if only characters are required.
  • All numbers – numeric check.
  • Floating numbers – e.g. input of currencies.
  • Letters and numbers.
  • String length – a specific length that cannot be exceeded.
  • Email validation.
  • Date validation.
  • Phone number validation.
  • Credit Card number validation.
  • Password.
  • IP address.

Error messages

If validation fails, the system should let the users know it by providing a clear and unambiguous message (usually one or two sentences) and ways to correct errors. Since users need to notice an error message immediately, it is a good practice to position it at the top of a web form, before all the other fields. This will also allow screen readers to easily access the message.

The message should be shown in red and contain an appropriate icon in order to get more attention. Icon should be globally recognizable, exclamation mark. This will also help people with visual impairments identify the meaning of the message. In addition to this, users should be informed about which text inputs need to be checked.

Apart from the error message and a list of invalid fields, the system should clearly mark fields that are invalid. This can be by:

  • Providing red messages or markers under every invalid field
  • Changing the background colour and border colour of invalid fields
  • Changing the colour of field labels

If you provide error tips or help, be short and informative.

Do's

  • Remove unnecessary inputs before an error can be made.
  • Provide a clear path to solve and complete form.
  • Enable smart defaults.
  • Provide hints on correcting errors.
  • Be flexible and forgiving with the data input.
  • Consider inline validation as soon as information is provided and disable dependent text inputs.
  • Display hints to avoid errors being made.
  • Auto-format when possible.
  • Report errors early.
  • Use role="alert" with aria-live="assertive" or aria-live="polite". Assertive will be pushed immediately to the screen reader as soon as the element appears while Polite will wait until the screen reader is done with its current task. It is a good practice to add it on the title of your error summaries.
